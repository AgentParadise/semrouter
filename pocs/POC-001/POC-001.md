# POC-001: Minimal Brute-Force Router

**Phase**: 1  
**Status**: Complete  
**Date**: 2026-05-06

## Goal

Answer the question from the spec:

> Given a local file of route examples, can I classify a new input into the right route with useful confidence and explainable candidate scores?

## What Was Proved

1. A keyword-based mock embedder is sufficient to prove the routing logic — real embeddings are not needed to validate the architecture.
2. Top-K per route averaging (k=3) correctly surfaces the best matching route even with a sparse vocabulary model.
3. Margin-based confidence detection works correctly — ambiguous inputs (e.g. "summarize and save") show low margin between top two routes.
4. The spec's JSON output format is fully implemented and the spec's decision status taxonomy maps cleanly to Rust enums.
5. High-risk route detection (pattern-matching on route name) correctly flags `unsafe_or_high_risk` routes as `requires_confirmation`.

## Approach

### Embedder

Used a 64-dimension keyword-bag embedder (`MockEmbedder`):

- 4 concept clusters × 16 keywords each: `coding`, `second_brain`, `research`, `model_routing`
- Each word in input text that matches a vocabulary keyword increments the corresponding dimension
- Small FNV-hash-based noise added so identical-zero vectors are differentiated
- Vector normalized to unit length at the end — cosine similarity becomes a dot product

This is not semantic in the neural-network sense, but it is sufficient to prove that the routing pipeline is correct.

### Scoring

1. Embed the input text
2. Compare to all 42 example embeddings using cosine similarity (dot product of normalized vectors)
3. Group similarities by route
4. For each route, take the top-3 similarities and average them
5. Sort routes by their average top-3 score

### Thresholds

| Setting | Mock value | Real embedder value |
|---|---|---|
| `minimum_score` | 0.25 | 0.72 |
| `minimum_margin` | 0.04 | 0.06 |

The lower thresholds are required for the mock embedder, which tops out at ~0.55 for well-matched inputs. Real embedding models produce similarity scores in the 0.75–0.95 range for semantically related texts.

## Test Results

### Unit Tests (4/4 pass)

- Mock embedder is deterministic for same input
- Output is a unit vector (norm ≈ 1.0)
- Coding texts are more similar to each other than to second-brain texts

### Integration Tests (6/6 pass)

- Coding input → `coding` route
- Second brain input → `second_brain_capture` route
- Research input → `research` route
- Candidates are sorted by score (high → low)
- Confidence fields (top_score, second_score, margin) are consistent
- High-risk routes trigger `requires_confirmation` policy

## Routes Included

13 routes covering all three primary use cases from the spec:

**Agent routing**: `coding_agent`, `research_agent`, `devops_agent`  
**Model routing**: `model_routing_high_reasoning`, `model_routing_cheap`  
**Second-brain**: `second_brain_capture`, `second_brain_retrieval`  

Plus: `coding`, `research`, `summarization`, `strategy_planning`, `needs_clarification`, `unsafe_or_high_risk`

## What Was Intentionally Omitted

Per Phase 1 spec:

- No `build` command / binary index (Phase 2)
- No decision logging to `decisions.jsonl` (Phase 3)
- No feedback command (Phase 3)
- No real embedding provider (Phase 4)
- No HTTP service (Phase 5)
- No hard negatives scoring (Phase 6)
- No ANN index (Phase 6)

## Key Insight Validated

The routing pipeline architecture is clean and correct. Each module (`embedding`, `scoring`, `decision`, `storage`) is independent and testable. Adding a real embedding provider in Phase 4 requires only implementing the `EmbeddingProvider` trait — nothing else changes.
